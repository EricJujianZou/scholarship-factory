# GH-12 — Deterministic date-parse layer (S6, date half)

## What shipped

A deterministic, offline parse layer that turns the verbatim `deadline`
string stored on an `Opportunity` into typed `date` values, without any
LLM or network call. It resolves relative expressions ("closes Friday")
against the stored anchor rather than the system clock, keeps every date
in a multi-deadline string uncollapsed, and falls back to `None` /
provenance `none` whenever a value can't be resolved — never guessing.

## Surface changes

### CLI / workflows

Not applicable — this is a library layer, no CLI/automation surface.

New public functions (`scholarship_factory/parse_dates.py`, exported via
`scholarship_factory/__init__.py`):

- `parse_deadline_dates(text: str | None, anchor: date) -> list[date] | None`
  — parses a deadline string against `anchor` using
  `dateparser.search.search_dates` (`languages=["en"]`,
  `RELATIVE_BASE=anchor`, `PREFER_DATES_FROM="future"`,
  `RETURN_AS_TIMEZONE_AWARE=False`). Returns every distinct date found, in
  first-seen order, or `None` if the text is blank or nothing matches.
- `typed_deadlines(opp: Opportunity) -> tuple[list[date] | None, Provenance]`
  — derives an anchor from `opp.source_observed_date`, falling back to
  `opp.first_seen` (via a tolerant `_parse_anchor` helper: ISO parse
  first, `dateparser.parse` as fallback), then calls
  `parse_deadline_dates` on `opp.deadline`. Returns `(dates, DERIVED)` on
  success or `(None, NONE)` otherwise. Read-only — does not mutate the
  store or the `Opportunity`.

## Behavior & breaking changes

None — both functions are new; nothing existing changed shape.

## How it was verified

All five ACs are covered by named tests in `tests/test_parse_dates.py`
(full suite: 31 passed, `uv run pytest -q`):

- Multi-deadline uncollapsed → `test_multi_deadline_not_collapsed`
  asserts `"June 1st, and October 1st"` + 2024-01-01 anchor produces
  `[date(2024, 6, 1), date(2024, 10, 1)]`.
- Relative resolves against anchor, not today →
  `test_relative_expression_resolves_against_anchor_not_today` runs
  `"closes Friday"` against two different anchors (2024-01-01 and
  2024-06-01) and asserts the results differ (Jan 5 vs Jun 7), driven
  entirely by `RELATIVE_BASE`.
- Unresolvable → `None`; computed → `derived`, else `none` →
  `test_unparseable_or_absent_returns_none` covers the `None` return
  path; `test_typed_deadlines_derived_provenance` and
  `test_typed_deadlines_none_provenance_when_unresolvable` /
  `..._when_deadline_absent` cover the provenance mapping.
- Deterministic + offline, all four case types covered → same test file;
  only `dateparser` is used, no network/LLM.
- `uv run pytest -q` green → test-stage run recorded exit code 0, 31
  passed.

## Review notes

- `search_dates` can over-match on very noisy deadline text; current AC
  inputs parse cleanly, but future noisy input may need result
  filtering. Do not fall back to comma-splitting — it would break
  single dates like `"September 15, 2024"`.
- `typed_deadlines` only reads from `Opportunity`; it does not write
  back, so the model's `*_source` validator (which only binds on stored
  provenance fields) doesn't apply here.
