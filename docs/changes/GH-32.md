# GH-32 ā€” Fetch: per-host politeness (rate limit + robots.txt)

## What shipped

Session 3 (Fetch) third link: `PoliteFetcher` in
`scholarship_factory/polite.py`, a thin wrapper over `fetch_url` (GH-26)
adding two good-citizen protections without changing its signature or
return type ā€” per-host minimum spacing between requests, and a cached,
fail-open robots.txt check per host. Nothing else is wired up; callers
opt in by using the polite layer (the pipeline ticket composes it).

## Surface changes

### CLI / workflows

None ā€” library class only.

New public class `PoliteFetcher` (`scholarship_factory/polite.py:29`),
re-exported from `scholarship_factory/__init__.py` alongside the new
constant `DEFAULT_MIN_INTERVAL = 2.0`.

- `PoliteFetcher(*, min_interval=2.0, timeout=..., user_agent=..., transport=None, clock=time.monotonic, sleep=time.sleep, fetch_fn=fetch_url)` ā€” clock, sleep, transport, and the underlying fetch function are all constructor-injected, so tests never sleep for real.
- `fetch(url) -> FetchResult` ā€” consults the host's robots.txt first (returning early on disallow, before any spacing wait, since no request is made), then applies per-host spacing, then delegates to `fetch_fn`.
- Per-host state (`_last_request`, `_robots`) is held in two in-memory dicts keyed by lowercased `netloc`; no persistence, no global state ā€” separate `PoliteFetcher` instances don't share history.
- robots.txt is fetched through the same injected `fetch_fn` (reusing `fetch_url`'s retry/User-Agent/transport path) and cached per host, including a cached `None` on fail-open, so it is requested at most once per host per instance.

### Schemas & data

No changes to `FetchResult` (merged in GH-11) ā€” a robots-denied fetch
returns the existing shape: `status_code=None`, `body=None`,
`error=f"robots.txt disallows {url}"`, which already yields `ok is False`
via `FetchResult`'s computed field.

## Review notes

- **robots.txt participates in rate limiting** ā€” it's a real request to
  the host, so the first fetch to any host pays a full `min_interval`
  sleep between the robots.txt request and the target fetch. This is
  deliberate (exempting it would mean back-to-back hits on first
  contact) but shapes runtime once a run touches many hosts.
- **Fail open covers 401/403 on robots.txt too**, not just 404/timeout ā€”
  per the locked design decision that politeness is insurance, not a
  gate that can wedge a run.
- Two non-blocking test nits from review, left as-is: `tests/test_polite.py:22` defines `_allow_all_robots` but never calls it (dead code); `tests/test_polite.py:67` reaches into the private `_robots` dict to pre-seed the cache, coupling that test to internals.
