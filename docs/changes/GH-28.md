# GH-28 — Rank: deterministic eligibility filter + sort vs applicant profile

## What shipped

A new `scholarship_factory/rank.py` module ranks stored `Opportunity` records against the single `ApplicantProfile`: deterministic hard-filter verdicts (`eligible | ineligible | expired`) with human-readable reasons, and a most-actionable-first sort. No LLM call, no fit-score float — matching the locked owner UX decision.

## Schemas & data

- `Verdict` (str enum) — `eligible`, `ineligible`, `expired`.
- `RankedOpportunity` (Pydantic model) — wraps an `Opportunity` with `verdict`, `reasons: list[str]`, parsed `deadline: date | None`, and parsed `reward: MoneyValue | None`.
- `RankedResults` (Pydantic model) — `eligible: list[RankedOpportunity]` (sorted) and `excluded: list[RankedOpportunity]` (ineligible + expired, unsorted).

## CLI / workflows

`rank(opportunities: list[Opportunity], profile: ApplicantProfile, *, today: date | None = None) -> RankedResults` — pure function, `today` injectable for tests, defaults to `date.today()`.

Internals:
- `_mismatch_reasons` — checks `opp.requirements` + `opp.description` text against two small explicit rule tables (`_REGION_CONSTRAINTS`, `_EDUCATION_CONSTRAINTS`), each a `(regex, human label, satisfied-by token set)` tuple. A rule only fires when the profile field also canonicalizes to a known alias (`_REGION_ALIASES`, `_EDUCATION_ALIASES`) — an unrecognized or null profile field can never trigger exclusion, satisfying "unknown is never ineligible."
- `_effective_deadline` — uses `typed_deadlines` (existing `parse_dates` layer); if any parsed date is `>= today` returns the soonest such date as not-expired, otherwise returns the latest past date as expired. No parseable date returns `(None, False)`.
- `rank` — per opportunity, computes mismatch reasons, deadline/expiry, and reward (via `typed_reward`); routes to `excluded` (ineligible or expired) or `eligible`, then sorts `eligible` by `(undated-last, deadline ascending, reward descending, title)`.

## How it was verified

`uv run pytest -q` reported 99 passed (21 in `tests/test_rank.py`), per the test stage. Acceptance-criteria mapping (from test + review stage evidence):

| Criterion | Evidence |
|---|---|
| Explicit region mismatch → `ineligible`, reason names both values | `test_region_mismatch_is_ineligible_with_reason`, `test_region_rule_fires_on_explicit_constraint` (4 cases) |
| Missing deadline/reward/requirements → stays `eligible`, sorts after dated | `test_missing_fields_eligible_and_sorts_last` |
| Past deadline → `expired`, excluded from eligible ordering but present with reason | `test_past_deadline_is_expired_and_excluded` |
| Two eligible dated opportunities → soonest deadline first, tie → larger reward first | `test_sort_deadline_then_reward` |
| All-null profile → nothing excluded | `test_null_profile_excludes_nothing`, `test_unrecognized_region_never_excludes` |
| Pure functions, no network/LLM, suite green | no I/O in `rank.py`; 99 passed reported by test stage |

Additional regression coverage added during review iteration 2: `test_benign_us_prose_does_not_exclude` plus per-rule benign-text negative tests (`test_region_rule_does_not_fire_on_benign_text`, `test_education_rule_does_not_fire_on_benign_text`) guard against false-positive substring matches.

## Review notes

- Iteration 1 review found a regex precedence bug: the `US` region pattern (`\bUS|United States (residents|citizens)`) let the bare `\bUS` branch match inside "using", "usually", "us in the lab" under `re.I`, falsely excluding eligible opportunities. Fixed by hoisting the required noun outside the alternation for all seven patterns: `\b(?:US|United States) (?:residents?|citizens?)\b`. The reviewer confirmed the fix generalizes the shape across every rule so the same bug class can't recur, and cross-checked the test count (21 = 6 original + 1 regression + 4 + 4 + 3 + 3 parametrized) against the file.
- Accepted limitation (not fixed): under `re.I`, "US residents" still matches contrived pronoun text like "join us residents" — judged not worth a per-rule casing flag for v1.
- Non-blocking observation: one benign-text education test case (`education_level="undergraduate"` against text mentioning "undergraduate research") is vacuous — the profile would satisfy that rule even if the pattern wrongly fired, so it can't actually fail. The other two education negative cases do genuinely exercise their patterns.
- Rule tables are intentionally small (v1) and two-sided (alias lookup on both the opportunity constraint and the profile field) — extending them should preserve this shape rather than adding raw-string fallbacks, which would reintroduce false-exclusion risk.
- Review sign-off could not independently execute `uv run pytest -q` (sandbox denial); the "99 passed" figure is inherited from the implement/test stages, corroborated by an independent test-count cross-check.
