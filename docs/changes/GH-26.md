# GH-26 â€” Fetch: static open-web fetcher

## What shipped

Session 3 (Fetch) first link: `fetch_url(url) -> FetchResult` in
`scholarship_factory/fetch.py`, a single static `httpx` GET with a
browser-like User-Agent and bounded exponential-backoff retry via
`tenacity`. It produces the `FetchResult` contract merged in GH-11,
representing every outcome (success, 4xx, retried 429/5xx, persistent
transport failure) honestly rather than raising. No adapters, robots
handling, caching, or headless rendering â€” those are separate tickets.

## Surface changes

### CLI / workflows

None â€” library function only.

New public function `fetch_url(url, *, timeout=15.0, user_agent=<Chrome UA>, transport=None) -> FetchResult` (`scholarship_factory/fetch.py:64`), re-exported from `scholarship_factory/__init__.py`.

- Single `httpx.Client` request per call, `follow_redirects=True`, browser-like `User-Agent` header by default.
- Retry via `tenacity.Retrying`, bounded at `RETRY_ATTEMPTS = 3` with exponential backoff (`RETRY_WAIT_MULTIPLIER = 0.5`), triggered on HTTP 429/5xx (`_is_retryable_status`) or `httpx.TransportError`. 4xx other than 429 never matches the retry predicate, so it fails on the first attempt.
- `transport` is a keyword-only injection point (`httpx.BaseTransport`) used by tests to run offline via `httpx.MockTransport`; defaults to real network when unset.

### Schemas & data

No changes to `FetchResult` itself (merged in GH-11) â€” this ticket only adds the producer function.

## Behavior & breaking changes

`httpx` moves from a dev-only test dependency to a runtime dependency in
`pyproject.toml`, since `fetch.py` now imports it at module scope. `tenacity`
is added as a new runtime dependency. `uv.lock` regenerated accordingly.

## Review notes

- The persistent-429 exhaustion path (`tenacity` raises `RetryError` on a
  retryable *result* rather than an exception; recovered via
  `exc.last_attempt.result()` at `fetch.py:79-80`) is implemented and was
  traced as correct during review, but has no dedicated test â€” only the
  persistent-transport-error path is covered. A cheap follow-up test
  asserting `status_code == 429` / `ok is False` after exhausting retries
  would pin this down.
- Retries do not honor a 429 response's `Retry-After` header; per-host
  politeness/robots handling is out of scope for this ticket.
- `transport` is injected as a keyword-only `httpx.BaseTransport` rather
  than accepting a full `httpx.Client`, so callers can't override
  `follow_redirects` or the User-Agent â€” both are behaviors this ticket
  exists to guarantee.
