# GH-42 — Identity: cross-source dedup

## What shipped

Session 5 (Identity). The v1 placeholder — opportunity identity ==
normalized `apply_url` (a `UNIQUE` index, GH-1) — is joined by a
secondary identity match so the same opportunity arriving from a listing
thin item and a detail page, or from two sources with different tracking
URLs, merges into one row instead of two. The match stays boring by
design: URL equality first, then exact normalized `title` + `organization`
equality. No fuzzy matching (edit distance, embeddings) — a missed merge
is acceptable, a false merge is not.

## Schemas & data

No schema changes. `scholarship_factory/identity.py` (new module) adds:

- `normalize_text(value) -> str | None` — lowercases, folds punctuation to
  whitespace, collapses whitespace; returns `None` for empty/None input.
- `find_duplicate(store, opportunity) -> Opportunity | None` — scans
  `store.list()`; returns a normalized-`apply_url` match immediately if
  found, otherwise falls back to the first row with matching normalized
  `title` + `organization` (both must be non-null on both sides).
- `merge_into(existing, incoming) -> Opportunity` — field-wise "richer
  wins": for each fact triple (`deadline`/`reward`/`cost` + their
  provenance/source companions) and each fill field (`organization`,
  `requirements`, `type`, `description`, `source_observed_date`), fills
  from `incoming` only where `existing` is `None`. Never overwrites an
  existing non-null fact. Returns `existing.model_copy(update=...)`, so
  `id` and `first_seen` are preserved.

## Behavior & breaking changes

`OpportunityStore.insert()` (`scholarship_factory/store.py`) now consults
`find_duplicate` before the URL-`UNIQUE`-index insert path: on a hit it
calls `self.update(merge_into(existing, opp))` instead of inserting a new
row. `last_seen` is refreshed via the existing `update()` path. Callers
that relied on `insert()` always creating a new row for same-title/org
records from different hosts will now see them merged.

## How it was verified

From `tests/test_identity.py`, run via `uv run pytest -q` (131 passed):

- "Different `apply_url`s (tracking params vs clean) still dedup" →
  `test_url_dedup_still_merges_one_row`: `len(store.list()) == 1`.
- "Different hosts, identical normalized title+org → one row, union of
  facts, `last_seen` refreshed, no overwrite" →
  `test_secondary_match_merges_union_of_facts`: `merged.reward == "$1000"`
  (existing kept over incoming `$2000`), `merged.deadline == "2026-07-01"`
  (filled from incoming), `merged.id == first.id`,
  `merged.first_seen == first.first_seen`, `merged.last_seen > first.last_seen`.
- "Same title, different org → two rows; same org, different title → two
  rows" → `test_same_title_different_org_does_not_merge`,
  `test_same_org_different_title_does_not_merge`: both assert
  `len(store.list()) == 2`.
- "Thin listing then rich detail (same normalized `apply_url`) → one row
  carrying detail facts" →
  `test_thin_then_detail_same_url_carries_detail_facts`: merged row keeps
  the thin listing's `title` but carries the detail's `deadline` /
  `deadline_source`.
- `merge_into` never overwrites an existing non-null fact →
  `test_merge_into_keeps_existing_non_null_fact`.

## Review notes

- `find_duplicate` does a full `store.list()` scan per insert — fine at
  current scale (no retroactive dedup sweep, per ticket scope), would need
  an index if the store grows large.
- Only the *first* title+org match found during the scan is merged into;
  if duplicate rows already exist under the old URL-only identity, this
  does not consolidate them (explicitly out of scope: "no retroactive
  dedup sweep of existing rows").

## File map

- `scholarship_factory/identity.py` — new: `normalize_text`,
  `find_duplicate`, `merge_into`.
- `scholarship_factory/store.py` — `insert()` consults `find_duplicate`

  `merge_into`.
- `tests/test_identity.py` — new: covers all five acceptance criteria plus
