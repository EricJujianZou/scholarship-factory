# GH-34 — Pipeline: sourcing run (seeds → adapters → fetch → extract → store)

## What shipped

The first end-to-end sourcing run, wiring the already-merged pieces (seeds,
adapters, static fetcher, both extract paths, store, CLI) into one function
and one CLI command. No new judgment logic: fetch failures are recorded
rather than raised, and both extract paths run on every successfully
fetched target.

## Surface changes

### CLI / workflows

`sf source --seeds seeds.toml [--db PATH]` (`scholarship_factory/cli.py`)
runs a sourcing pass over a seeds file and writes results to the opportunity
store. Prints targets attempted, opportunities stored, skipped seeds (with
reason), and any fetch failures (with status/error); always exits 0, since a
run that reports its failures honestly still succeeded at running. The CLI
module docstring no longer claims the tool is read-only, since `source`
writes.

### Schemas & data

New in `scholarship_factory/pipeline.py`:

- `run_sourcing(seeds, store, *, fetch_fn=fetch_url, extract_fn=extract, jsonld_fn=extract_jsonld) -> SourcingReport` — plans targets via `targets_for_seeds`, fetches each target, and on a successful fetch runs both the JSON-LD structured path and the LLM path against the body, upserting every resulting `Opportunity` into the store.
- `TargetOutcome` (pydantic model): `url`, `ok`, `status_code`, `error`, `opportunities_stored` — one per attempted target.
- `SourcingReport` (pydantic model): `outcomes: list[TargetOutcome]`, `skipped: list[SkippedSeed]`, plus computed fields `targets_attempted` (`len(outcomes)`) and `opportunities_stored` (sum across outcomes) — derived rather than tracked separately, so the counts can't drift from the per-target detail.

`run_sourcing`, `SourcingReport`, and `TargetOutcome` are exported from
`scholarship_factory/__init__.py`.

## Behavior & breaking changes

`sf`'s docstring previously described the CLI as read-only/no-network; that
is no longer accurate now that `source` exists, and the docstring was
updated accordingly. No existing command's behavior changed.

## How it was verified

All tests in `tests/test_pipeline.py` and `tests/test_cli.py` use injected
fetch/extract callables — no network, no real LLM.

- *url + instagram seed → 1 target, skipped-unsupported, opps stored* → `tests/test_pipeline.py::test_url_and_skipped_seed_reports_one_target_and_stores_opportunities`: asserts `targets_attempted == 1`, `skipped[0].reason == SkipReason.UNSUPPORTED`, and the stubbed opportunity's `apply_url` is in `store.list()`.
- *Fetch failure recorded, other targets still process, nothing raises* → `tests/test_pipeline.py::test_fetch_failure_recorded_other_targets_still_process`: one target's fetch returns `status_code=403`, and its outcome has `ok is False`, `status_code == 403`, `error == "forbidden"`, while the sibling target's opportunity still lands in the store.
- *Re-run doesn't duplicate; `last_seen` refreshes* → `tests/test_pipeline.py::test_rerun_against_same_store_does_not_duplicate`: calling `run_sourcing` twice against the same store leaves `len(store.list()) == 1`, with the same `id` and `first_seen`, and `last_seen` on the second run `>=` the first.
- *Both extract paths run, union stored* → `tests/test_pipeline.py::test_both_extract_paths_run_on_lablab_fixture`: real `extract_jsonld` against the lablab fixture plus a recording extract stub returning a distinct-`apply_url` prose record; both are called once, and `store.list()` contains both apply URLs.
- *`sf source` prints summary and exits 0* → `tests/test_cli.py::test_source_prints_summary`: `main(["source", "--seeds", ..., "--db", ...])` returns `0`, and stdout contains `targets attempted: 1` and `skipped: 1`.
- *All tests offline* → every network-touching callable (`fetch_fn`, `extract_fn`, `jsonld_fn`) is injected in tests; `extract.py` only imports `anthropic` inside its lazy default-client path, so importing `cli`/`pipeline` never reaches the SDK.

Full suite: 115 passed.

## Review notes

- **Field-shadowing on collision (deferred, not a bug):** the store's
  conflict clause updates only `last_seen` on upsert, so if the JSON-LD path
  and the LLM path ever yield the same normalized `apply_url`, whichever
  inserts first wins every other field. JSON-LD runs first and carries no
  deadline/reward, so a collision would let the thinner record win. Fixing
  this needs a merge/field-preference policy, which is out of scope for this
  ticket (no new judgment logic) and is left for the refresh ticket.
- Relatedly, `SourcingReport.opportunities_stored` counts opportunities
  *extracted*, not distinct rows written — an intra-run `apply_url`
  collision between the two paths would over-report by one. Same deferral
  as above.
- Extractor exceptions are not caught; only fetch failures are. This is
  deliberate per the ticket's locked decision — a blanket catch around the
  extractors would swallow real bugs.
- `source_url` for extracted records is `result.final_url` (post-redirect),
  while `TargetOutcome.url` keeps the originally requested `target.url` so
  the report still maps back to the adapter plan.
